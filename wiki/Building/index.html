Building individual components

While the system as a whole makes use of several build tools (Maven, Gradle, Autotools, etc.), the way of connecting the individual components is completely Maven based, which means that upon releasing a certain component, the artifact is made available (“deployed”) in an online Maven repository to any component that wants to make use of it. Dependencies are resolved and downloaded automatically and do not have to be copied explicitely.

Gradle is Maven compatible. Liblouis and Libhyphen are built with Autotools, but the binary artifacts are made part of the Maven system by packaging them into a special kind of JAR file (see e.g. [liblouis-nar][]).

As a result of this approach, components only need to be built locally when changes have been made to them and need to be tested or used in other components. These development versions must be marked as “snapshots”. Although theoretically possible, it does not make a lot of sense to rebuild artifacts that have already been released. Maven has a rule that says versions must always be snapshots except in a “release commit”, and that no two release commits may have the same version number. Because when under development an artifact (snapshot) can change from one day to the other, no released versions should ever depend on a snapshot. Snapshot dependencies between components that live in different source repositories are also best kept to a minimum.

The various components are listed below in the order they need to be built.

liblouis

Liblouis is the component that implements the braille translation codes for most of the languages, specified in so called Liblouis translation tables.

For more information and build instructions see liblouis/liblouis and liblouis/liblouis-nar.

dotify

Dotify is the component that implements the Open Braille Formatting Language OBFL, in other words it is our formatter. Dotify also provides an implementation of the Swedish braille code.

For more information and build instructions see joeha480/dotify.

braille-css

braille-css is the ANTLR based parser for Braille CSS.

For more information and build instructions see snaekobbi/braille-css.

brailleutils

Braille Utils contains everything related to converting from and to the Portable Embosser Format PEF.

For more information and build instructions see joeha480/brailleutils.

pipeline-mod-braille

pipeline-mod-braille is the component (group of Pipeline modules) that builds on the Pipeline framework and integrates and interconnects all the lower-level components such as dotify, liblouis, braille-css, etc. It does not include agency specific things (see below).

pipeline-mod-braille is special in that it is a multi-module project. The modules have inter-dependencies as well as external dependencies. All modules can be built at once by invoking mvn from the root directory of the project (the “aggregator”). The build order is determined by Maven. If only a single module has changes since it was last built, it suffices to rebuild that single module.

In order to avoid having to release all modules whenever a single module is released, every module is disabled after each release (by removing it from the aggregator), and modules are enabled again as soon as they have changes in the code. Important is that whenever a module is enabled and you want to use that snapshot elsewhere, you should also update that module’s version number in bom/pom.xml, the so called “Bill of Materials”. The purpose of the BoM is to be a central record of the version numbers of all modules and external dependencies that can be imported by every module and every external project that wishes to use a braille module or the braille modules as a whole.

The version of the braille modules as a group is declared in both parent/pom.xml and bom/pom.xml and is chosen to match the version of the next pipeline-assembly.

Dependencies

  • liblouis
  • dotify
  • braille-css
  • brailleutils
  • libhyphen

The versions of these external dependencies are declared in bom/pom.xml.

Build instructions

mvn clean install

This will also invoke the unit tests. Java 8 is required.

pipeline-assembly

pipeline-assembly is the build script that assembles the complete Pipeline engine including the braille modules, and packages everything in a DEB file.

The version is declared in pom.xml.

Dependencies

  • pipeline-mod-braille

The version of braille-modules-bom in pom.xml must match the version of pipeline-mod-braille, and all modules and dependencies (including transitive ones) from pipeline-mod-braille should be included in copy-pipeline-bundles resp. copy-libs-bundles.

Build instructions

mvn clean install -Pdeb

If while developing there is the need to test a component with the web interface, it may be convenient to build a “development version” of the Pipeline engine which automatically and dynamically reloads bundles upon installing them in your local Maven repository (with mvn install), allowing you to skip the step “Building the entire system” after each change in the component or one of its dependencies.

mvn clean package -Pdev-launcher

Until a certain bug is fixed an additional step is required: if you are on Linux run rm target/dev-launcher/etc/*{mac,windows}* (and something similar if you are on Windows).

Launch with:

target/dev-launcher/bin/pipeline2

Java 8 is required.

To check if it works, go to e.g. “http://localhost:8181/ws/alive” or “http://localhost:8181/ws/scripts” in your browser.

Note that whenever you want to include a new component, the dev-launcher needs to be rebuilt.

pipeline-mod- celia | dedicon | mtm | nlb | nota | sbs

These modules contain all agency specific scripts, code, CSS files, liblouis tables, etc. that do not have a place in the main braille modules.

The version is declared in pom.xml.

Dependencies

  • pipeline-mod-braille

The version of braille-modules-parent in pom.xml must match the version of pipeline-mod-braille.

Build instructions

mvn clean install

This will also invoke the unit tests.

pipeline-webui

This is the web-based user interface.

For build instructions see Building and hacking.

Building the entire system including the web interface

system

#### Dependencies - pipeline-assembly - pipeline-mod-celia - pipeline-mod-dedicon - pipeline-mod-mtm - pipeline-mod-nlb - pipeline-mod-nota - pipeline-mod-sbs - pipeline-webui

The versions of the dependencies are declared in roles/test-server/vars/debs.yml.

Build instructions

To install the system locally (on a Linux machine): sh make ansible-playbook test-server.yml -l local

To install the system on a Vagrant test VM: sh vagrant up make ansible-playbook test-server.yml -l vagrant-test-server
